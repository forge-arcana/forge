# /pry — project-local learnings (awaiting fold)

## "Build From Source" Is Not a Malware Control — Pin + Review + Lock Is (2026-06-14)
**Learning**: When a supply-chain "should we build the dependency from source so we don't absorb malicious code?" claim surfaces, separate two assumptions people conflate: (a) *does the binary match the published source?* — solved by building from source (or reproducible-build attestation); (b) *is the source + its dependency tree trustworthy?* — the actual malware vector, NOT touched by compiling. Compiling is necessary-not-sufficient and it *expands* the trusted surface: `build.rs` and proc-macro deps execute arbitrary code at build time, and the real attack surface is the transitive lockfile (`Cargo.lock`/`package-lock`), not the project's own `src/`. The controls that actually stop malicious code are: pin an exact reviewed ref + record an artifact hash in YOUR repo (defeats post-review release swaps; same-origin checksums.txt is integrity not authenticity) + human delta-review on bump (covering lockfile + build scripts, not just src). "Refresh the dep every cycle" inverts safety — it auto-absorbs unreviewed commits; instead detect upstream drift and gate the bump behind the same review the project already uses for its own content.
**Apply when**: Any "vendor this / build from source / auto-update the dependency" proposal framed as a security improvement. Map it to the pin+hash+delta-review+lock model and ask which of (a)/(b) the proposer actually fears.

## Reimplement the Technique, Not the Tool — Scope to Your Actual Profile (2026-06-14)
**Learning**: "Steal the bits of $TOOL we need so we never maintain/review it" is the right reframe *only if* "the bits we need" is scoped tightly. Taken literally ("reimplement $TOOL") you inherit its whole maintenance surface. The unlock: a general-purpose tool's value is usually its *breadth* (rtk: 100+ command parsers, exit-code fidelity, edge cases), not its core technique (output compression = dedup/strip/truncate, ~80 lines of awk). Breadth is exactly what a single consumer doesn't need. Split the steal into two tiers: format-agnostic logic that never rots (own it forever, zero drift) + a tiny slice of format-aware logic only for the handful of cases you actually hit (grows lazily). This beats depending on the tool (no binary, no supply chain, no review gate) for the narrow case, at the cost of forgoing breadth + community-amortized maintenance. Owning bash logic also stays tool-neutral where a binary wouldn't.
**Apply when**: Evaluating a third-party tool whose surface dwarfs your use of it. Ask "what fraction of its commands/features do we invoke?" — if it's a handful, reimplementing the format-agnostic core + your tiny slice is often cheaper *over time* than the dependency's review/update tax.
**Forge-worthy**: yes

## Per-Session Token/Cost Monitoring Needs No Backend — Read the Transcripts (2026-06-14)
**Learning**: "Monitoring token burn per session requires OTEL + an observability backend + API keys" is FALSE for solo/team-of-one introspection. Claude Code writes full `message.usage` (input/output/cache-read/cache-create) + model + timestamp per assistant turn into `~/.claude/projects/<path-with-slashes-as-dashes>/<uuid>.jsonl`. A jq+awk script sums it per session and estimates cost with a local price table — zero deps. (ccusage and lean-ctx's dashboard both read these same files.) Gotchas: dedupe streaming-duplicate records by `message.id`; use top-level `usage`, not the `iterations[]` sub-array (double-counts); cache-read tokens dominate raw counts but are cheap, so report output tokens as the honest spend signal. OTEL is the *team* answer (cross-machine dashboards); it's overkill when the data is already on disk. Build the monitor FIRST so you can verify any later saving's real delta — vendor "60-90%" numbers are self-measured.
**Apply when**: Any "we need observability infra to see token/cost" claim, or before adopting a token-saving tool (measure the baseline first). Path-encoding + the dedup/iterations gotchas are the non-obvious parts.
**Forge-worthy**: yes

## Token-Saving Analyses Are Denominated in the Runtime's Scarcest Resource — Re-Rank When the Runtime Changes (2026-06-19)
**Learning**: A token-reduction lever list is implicitly priced in *one* currency — for a cloud API (Claude) that's per-token dollars, so the biggest file-read wins top the list. Port the same skill to a **local open-weight model** (GLM 5.2: 1M context, MIT, local deploy, two thinking-effort levels) and the currency flips to {context-window budget, wall-clock prefill+decode latency, single-GPU serialization} — marginal token cost ≈ 0. The *ranking inverts*: (a) fan-out subagents that ran cheap-and-parallel in the cloud now **serialize on one GPU** → fan-out batching becomes the headline latency ceiling, not a "spiky/optional" lever — and on a TUI with no subagent spawning the sequential fallback fires automatically, so an explicit sequential-fallback contract is the real fix; (b) file-read gating loses its dollar rationale AND (verify the number!) its window rationale if the local model also has a big context — GLM 5.2 is 1M, so the "smaller window crowds out work" argument I reached for was FALSE, leaving only prefill-latency; (c) output-compression (strip/dedup/truncate) is the one harness-agnostic win — helps tokens, prefill, and window simultaneously; (d) measurement tooling that reads the *cloud vendor's* transcripts (`/burn` → `~/.claude/.../*.jsonl`) doesn't exist on the local runtime — instrument wall-clock + the OpenAI-compatible `usage` block instead; (e) per-skill *model-tier* ceilings are inert on a single-model runtime, but the *intent* (cheap skills shouldn't burn max reasoning) ports onto an effort/thinking-level knob if the local model exposes one.
**Apply when**: Reviewing any token/cost optimization, or porting a fan-out-heavy skill repo to a local/self-hosted model. Ask "what's actually scarce in the TARGET runtime?" before accepting the lever ranking — and verify the target's context window before claiming file-reads crowd it. The cloud→local shift turns the parallelism assumption (cheap concurrent subagents) into its opposite (serialized GPU passes).
**Forge-worthy**: yes

## "Do I Still Need $VENDOR?" Splits Into Model / Harness / Subscription — Answer Them Separately (2026-06-19)
**Learning**: When a user asks "open model X beats $VENDOR's model, so my $VENDOR subscription is now pointless," the claim bundles three separable decisions: (a) **model quality** — is X actually ahead, or just *near*-parity? Benchmark marketing ("nears Opus", "0.7 pts from the king") usually means *close on short evals but the gap is WIDEST on long-horizon agentic/multi-step tool-use* — which is exactly where a skill stack's heavy arts (autonomous builders, multi-persona QA, go-live reviews) live, so near-parity on aggregate ≠ parity on YOUR workload; (b) **harness** — the agentic loop (skill discovery, subagent fan-out, hooks) is the vendor's *CLI*, not its *model*; you can keep the CLI and point it at the open model via a base-URL override + OpenAI↔vendor proxy, or move to a neutral harness the stack already supports; (c) **subscription/cost** — "drop the sub = self-host the open model free" is usually FALSE: frontier open weights are huge (e.g. 753B → hundreds of GB VRAM even quantized → $5–7k rig or quality-lossy 2-bit). The real cost-saver is the *hosted* open-model API or its flat coding sub (often 3–6× cheaper per token), NOT local self-host — reserve self-host for sovereignty/offline, not savings. The unlock: a tool-neutral stack means the answer is almost always "yes you can switch the model, keep the harness, go hosted-not-local — but A/B your own heaviest task first because that's where the open model's gap (if any) concentrates."
**Apply when**: Any "should I cancel $VENDOR / switch to open model X" question for an agentic tool stack. Decompose into model/harness/subscription, check whether the benchmark gap lands on the user's actual heavy workload, and steer cost-saving toward hosted-API/flat-sub over local self-host unless data-residency is the real driver. Recommend a cheap empirical A/B (point the existing harness at the hosted open model, run the real load) before any cancellation.
**Forge-worthy**: yes

## Forge's Biggest Token Lever Is the Fan-Out Multiplier, Not Bash Output (2026-06-14)
**Learning**: In a skill repo with fan-out arts (`/poke`, `/press`, `/pound`'s 21 personas, `/smith` apprentices), the evidence a shared script emits is paid for *once per subagent* — so every byte is multiplied ×N. Compressing the repo's own evidence scripts (strip repeated absolute-path prefixes, dedup matches, cap width *before* the head-cut so the cut carries more signal, keep match COUNTS exact since they're computed separately) is higher-leverage than any general-purpose bash-output compressor, because third-party tools (rtk/lean-ctx) compress bash output but never touch the subagent-prompt multiplier. Counter-rule: do NOT compress content fed to single-agent *triage* (e.g. fold-evidence's full file dumps) — truncation there hides entries; correctness over tokens.
**Apply when**: Optimizing token cost in any multi-agent orchestration. Find what's fed to the fan-out and compress that first; leave correctness-critical single-agent inputs full.
**Forge-worthy**: yes
