Forge
A simple guide for everyone

Forge is like a master recipe book for your AI assistant.
It gives Claude new recipes (skills), helps Claude remember what worked and what didn't, and keeps everything organised — so your AI gets better every single day.

🍳 The kitchen analogy

Imagine a master recipe book that all chefs share. Your kitchen (~/.claude/) holds your personal copy of the recipes. When you want something cooked (your project), Claude opens the kitchen, picks the right recipes, and gets to work. When you discover a better way to cook something, you write it back into the master book so every chef benefits.

Forge = the master recipe book (shared, always improving)
~/.claude/ = your personal kitchen — your tools and recipes (called "the membrane")
Your project = the meal you're currently cooking

The Core Loop
Three commands that run the whole system. Think of them as morning, afternoon, and evening routines.
🔍
/mark
Just Look Check what's in sync, what's drifted. Read-only. Never changes anything.
🍳 Like checking which recipes in your kitchen are newer than the master book
⬇️
/cast
Pour Skills In Stocks your kitchen with everything from the master recipe book. Updates skills, learnings, memory.
🍳 Like stocking your kitchen with the latest updated recipes from the master book
⬆️
/fold
Absorb Back Take lessons learned during work and fold them back into the forge. The system grows smarter.
🍳 Like writing your best discoveries back into the master recipe book
Important: Every transfer shows you a table first and asks for your approval. Nothing happens behind your back. You are always in control.
/mark
The Inspector
Like a hallmark stamp on jewellery — it inspects quality without changing anything
🍳 Kitchen analogy

Imagine opening your kitchen cupboard and comparing every recipe card to the master book — "this one is outdated, this one is missing, this one has been updated." You don't cook anything. You don't change anything. You just see exactly what's different and get a clear report.

🔎
What it checks
  • Are all skills up to date?
  • Are any skills missing?
  • Have learnings drifted?
  • Is memory in sync?
  • When was the last /cast?
📋
What it produces
A clean table showing every skill:
✅ IDENTICAL — in sync
⚠️ DIFFERS — needs update
➕ ADDED — new skill
➖ REMOVED — deleted
⚠️
When to use it
Before /cast, to see what will change.

When you're curious about the state of things but don't want to do anything yet.

Think of it as a dry run.
/cast
The Deployer
Pours molten metal from the forge into the mold — skills flow from forge into your Claude
🍳 Kitchen analogy

The master recipe book has been updated with better techniques and new recipes. /cast is the process of copying all those updates into your kitchen — swapping out old recipe cards for the fresh versions, adding any new ones that are missing. Your kitchen is now fully up to date.

📋 What /cast does — step by step
1
Pull latest forge Gets the newest version from GitHub. Like downloading today's newspaper.
2
Compare skills, learnings, memory Figures out what's new, what's changed, what's been removed.
3
Show you a table — ask for approval Presents every change and waits. Nothing happens until you say yes.
4
Deploy approved items Copies skills to ~/.claude/skills/, learnings to ~/.claude/learnings/, etc.
5
Record the baseline Saves a timestamp/commit SHA so future /mark and /fold can detect drift accurately.
💡 Run /cast whenever you update the forge repo, or on a fresh machine.
/fold
The Absorber
Layer experience back into the steel — what you learned today makes the forge smarter tomorrow
🍳 Kitchen analogy

You spent the day cooking (your project work). You discovered that adding lemon at the end makes the sauce far better. /fold is writing that discovery — as a margin note — back into the master recipe book, so every chef who uses it from now on automatically gets that improvement.

What gets absorbed?
  • Lessons marked "Forge-worthy: yes"
  • Team memories and conventions
  • Config settings that proved useful
How the quality gate works
1. /fold scans project memory files
2. Finds entries marked Forge-worthy
3. Strips all project-specific names
4. Shows you a table to approve
5. Only approved items get absorbed
The Self-Improving Loop:
Arts write learnings → /fold absorbs them → next time arts run, they READ those learnings → they get smarter. Every run makes the whole system better.

What Are Arts?
Special skills where Claude puts on a different chef's hat and thinks like a specialist
🍳 Kitchen analogy

Imagine a kitchen with specialist chefs you can call in. 🍣 Sushi chef → evaluates your dish's presentation. 🥐 Pastry chef → checks precision and structure. 🍷 Food critic → attacks it from every angle. Arts are Claude's collection of specialist chef hats — each bringing deep expertise and a completely different way of thinking.

⚡
Regular Skills
Just do a task.
"Commit this code."
"Show me the status."

Simple. Direct. No persona.
🎭
Arts (Special Skills)
Adopt an expert persona.
Think differently.
Write lessons after each run.

Get smarter over time.
Every art follows the same routine:
Pre-flight (read past lessons) → Do the work (as the expert) → Post-flight (write new lessons)

Arts: Creating & Planning

Use after: /prime — give it the blueprint, get structural feedback before building

Arts: Unstuck & Clean
Two special arts that don't fit the normal pattern
🔧
/pry
The Lever — Relentless Problem Solver
Use when you're completely stuck. Someone said "this can't be done." /pry refuses to accept that.
It wears 3 hats in sequence:
🎩 Skeptic — challenges the claim
⛏️ Prospector — digs for alternatives
🔄 Reframer — changes the question
🍳 Like the chef who refuses to say "we can't make that" — keeps trying substitutions and techniques until they find a way to get the dish on the table.
🧹
/purge
The Purist — Forge Cleaner
/purge doesn't work on your project. It works on the forge itself. Removes stale rules, duplicates, outdated knowledge.
Does NOT build. Does NOT review.
Only tends the forge — keeping it sharp and clean.
🍳 Like deep-cleaning the kitchen itself — doesn't cook, but without it the kitchen fills up with stale ingredients, cluttered shelves, and outdated recipes nobody uses. Done occasionally, not daily.
⚒️
The Smith
The master of the forge. The user's proxy. Uses ALL the arts to build entire products.
🍳 Kitchen analogy

Imagine a head chef who knows every technique, manages every specialist in the kitchen, coordinates all the courses running in parallel, won't let a dish leave until it's perfect — and calls in the food critics to test it before serving. You just describe what you want to eat. The head chef handles the entire kitchen.

🔥
Iterative Heats
Works in cycles: plan → build → evaluate → fix. Repeats until perfect.
👥
Summons Apprentices
Creates sub-agents to work in parallel. Never does things one-by-one when they can run at the same time.
🎯
Zero Tolerance Gate
Won't stop until zero critical issues remain. Runs /pound at the end. True quality gate.
How to use: Give /smith a probed blueprint (from /prime + /probe). It handles everything else — autonomously — from first commit to shipped product.

The Self-Improving Loop
How Forge gets smarter every single time you use it
🍳 Kitchen analogy

Every chef writes margin notes after each service: "The sauce needed more acid" or "This timing worked perfectly." Those notes get written into the master recipe book. Every chef who cooks from it next time is starting from a better foundation than the chef before. The kitchen never stops improving.

1
Art runs on your project e.g., /poke reviews your code. It notices patterns, good and bad.
2
Art writes to memory/poke-learnings.md Records lessons like "Forge-worthy: yes — always validate at API boundary, not just frontend."
3
/fold scans and promotes Finds Forge-worthy entries, strips project-specific details, promotes to ~/.claude/learnings/general.md.
4
/fold absorbs into forge The universal lesson enters forge/learnings/ permanently. Now every project benefits from it.
5
Next art run reads learnings (pre-flight) The art is now smarter than before. Every run feeds the next. Compounding improvement.

The Hard Rules
Rules that are never broken. These keep the system safe and trustworthy.
🚫
No Auto-Commit
Claude never commits code automatically. Always asks first. You are in charge of what gets saved.
🍳 Like asking the head chef before adding anything permanent to the master recipe book
🔗
No Command Chaining
Never runs multiple bash commands together with && or ; — each command runs alone. Prevents accidental disasters.
🍳 Like finishing one dish completely before starting the next — no chaos in the kitchen
🔒
Only /fold Writes to Forge
No direct edits to forge's learnings or memory from a project. /fold is the only gateway — with your review.
🍳 Like only the head chef can update the master recipe book — no one else writes in it directly
🏷️
No Project Names in Forge
Forge is shared across all projects. Learnings must be universal. "MidWicket had a bug" → "Marketplaces need rate limiting."
🍳 Write "reduce heat after adding cream" — not "this worked at the Milan restaurant"
✨ The Golden Rule: Forge brings intelligence. You bring wisdom. Nothing changes without your approval.

One-Time vs Iterative
Some commands you run once and never think about again. Others you run constantly.
🍳 Kitchen analogy

One-time = Stocking your kitchen when you first move in (you do it once, then top up occasionally). Iterative = Tasting and adjusting seasoning every time you cook — it's part of every single meal.

1️⃣
One-Time (or Rare)
Run once when setting up, or very occasionally
/cast Fresh machine setup OR after forge updates
/prime Once per project idea — to create the blueprint
/probe Once per blueprint — validate before you build
/purge Occasional — clean forge when it feels cluttered
/srs Once per project — sets up restart scripts
/vsix Per release — publish VS Code extension
🔄
Iterative (Run Often)
Part of your daily or per-feature rhythm
/wawa Daily — start of day status check
/poke Every sprint or major PR — light code check
/qt After every fix — verify before handing back
/wrap Every commit — proper commit ritual
/ponci Every push — push + watch CI
/fold End of each project milestone — absorb learnings

Full Project Lifecycle Map
Every stage of building something — and exactly which commands belong there
💡
STAGE 1 — Idea You have a vague idea. Nothing is built yet.
/prime → Turns your fuzzy idea into a clear blueprint through guided questions one-time
🏛️
STAGE 2 — Blueprint Review You have a plan. Is it a good plan?
/probe → Senior architect tears apart the plan, finds gaps before you write code one-time per plan
/pitch → Business model check — is this worth building? before build + before ship
⚙️
STAGE 3 — Project Setup First time on a new codebase or fresh machine.
/cast → Deploys all forge skills + sets up CLAUDE.md conventions for the project one-time per machine/project
/srs → Creates restart.sh and kill-zombies.sh scripts for local dev one-time per project
🔨
STAGE 4 — Active Development You're coding every day. This is the main loop.
/wawa → Morning status: what's done, what's next, any blockers daily
/poke → Light code quality check — catches bad patterns early every sprint / PR
/preen → UI/UX design check — run whenever the interface changes per UI change
/qt → Quick test after every fix — did it actually work? after every fix
/wrap → Commit properly: lint → test → stage → commit with context every commit
/ponci → Push to GitHub and watch CI pass live every push
/pry → When completely stuck — use this to find any path forward when stuck
🚀
STAGE 5 — Pre-Launch / Release Before any milestone or shipping to users.
/press → Security, scalability, operations check. Fix everything it finds. before every milestone
/pound → 21 adversaries attack your code. The hardest test. Fix critical issues. before ship only
/pitch → Final business model sanity check before users see it before ship
/cicd → Full local pipeline: lint + typecheck + test + build + deploy. Auto-fixes. before deploy
🔁
STAGE 6 — After Milestones / Ongoing Keep forge healthy and growing smarter.
/fold → Absorb lessons from this work back into forge. Makes future projects smarter. end of each milestone
/mark → Check if forge and your Claude are in sync. Safe read-only inspection. anytime you're curious
/cast → Re-run when forge is updated by you or the team. Redeploy latest skills. after forge updates
