# /pry — accumulated learnings

## Open-Weight Model Choice Is Independent of Infrastructure Choice (2026-05-13)
**Learning**: When a "Provider X is in $REGION, so we can't use their model for $TARGET-REGION product" claim surfaces, decompose into two separable questions: (a) is the model open-weight (freely distributed weights anyone can serve)? (b) which infrastructure providers host that model in the target region? Open-weight model families are served by many infrastructure providers across many regions; choosing the host is a separate decision from choosing the model. Conflating the two produces false constraints that drop viable models from consideration.
**Apply when**: Reviewing a claim that a model can't be used due to provider sovereignty, data-residency, or jurisdiction concerns. Always ask whether the model is open-weight and whether alternative regional hosts exist before treating the constraint as binding.

## OpenAI-Compatible API Is the De Facto Inference Protocol (2026-05-13)
**Learning**: Most modern LLM hosts now expose the same `POST /v1/chat/completions` request/response shape regardless of underlying model. Adding a new provider adapter is therefore ~20 minutes of work: copy an existing OpenAI-shaped adapter, swap endpoint URL + auth header, done. Treat provider-adapter cost as cheap when estimating effort to add a new model path or evaluate a new vendor — the integration is not the bottleneck.
**Apply when**: Estimating effort to add a new model provider to a routing layer. If the provider exposes an OpenAI-compatible endpoint (most do), budget hours not days. Bias toward "try it" rather than "evaluate it" for adapter-only integrations.

## Major Cloud Providers Host Open-Weight Models in Constrained Regions (2026-05-13)
**Learning**: The hyperscaler ML platforms (AWS, GCP, Azure) now serve major open-weight model families from EU/UK datacenters and other residency-sensitive regions. For products with data-residency requirements, the question "can we use this open-weight model" changes from "no, the publisher is in $RESTRICTED_REGION" to "yes, via your existing cloud provider's region". The cloud provider becomes the residency anchor — not the model publisher. This is a structural shift that invalidates many "we can't use $MODEL because of $JURISDICTION" objections that were valid a year earlier.
**Apply when**: Auditing whether an open-weight model can be used in a residency-constrained product. Check the major cloud ML platforms' regional model catalogs before concluding the model is off-limits.

## Router-Middleman Aggregators Take a Margin Slice (2026-05-13)
**Learning**: API-aggregator services that proxy requests across many LLM providers add a per-call markup or commission. Even when they offer regional routing or unified billing, they sit as a margin-bearing layer between the consuming product and the actual host. If unit-economics discipline matters, router middlemen are off the table by design — the choice becomes either direct-to-host integration (own the keys, own the margin) or accept the aggregator's permanent COGS slice. Owned first-party gateways (your own multi-provider routing service hosted on your own infra) are a different category: no markup, but operational cost is yours. Distinguish the two: a router middleman is third-party and margin-bearing; a first-party gateway is owned and operational-only.
**Apply when**: Reviewing the inference-layer architecture and someone proposes "let's just route through $AGGREGATOR for simplicity." Check whether unit economics absorb the aggregator markup, and whether the simplicity gain justifies losing direct provider control (rate-limit visibility, failover policy, cost calc authority).

## "Can't Use $MODEL Because $HOST" Reframe Pattern (2026-05-13)
**Learning**: When a "we can't use $MODEL because $HOST has $PROBLEM (sovereignty / data-use / outage / pricing)" claim surfaces, the reframe is: "find another host with the same weights in the target region; if none exists, self-host (open-source serving runtimes on cloud GPU instances); if neither, drop the model not the use case." The model is not the same thing as the inference path. Treating them as separable concerns usually shrinks the constraint surface significantly.
**Apply when**: A model-choice debate stalls on a host-specific constraint. Restate as "what hosts serve these weights, and what are their respective constraints?" before accepting the constraint at face value.

## Transfer Mechanism vs Provider Data-Use Policy Are Different Concerns (2026-05-13)
**Learning**: Cross-border data flow has two distinct compliance gates that are easy to conflate: (a) is there a lawful transfer mechanism in place (Standard Contractual Clauses, adequacy decisions, processor-on-behalf-of-user framing)? AND (b) does the destination provider's default-tier data-use policy match the product's promise to users (e.g., "we will never use your data for training")? A clean transfer mechanism doesn't help if the destination provider's default terms put user inputs into training. Verify both per provider before claiming "sovereignty solved" or "residency handled."
**Apply when**: Reviewing a third-party inference provider for use in a product with data-protection promises. Check both the transfer mechanism AND the provider's published data-use policy at the tier the product will actually consume. Enterprise tiers often have stricter defaults than developer-tier APIs — confirm which tier is being procured.

## Verify "Well-Trodden Path" Claims Before Agreeing or Disagreeing (2026-05-13)
**Learning**: When a stakeholder argues "this is well-trodden territory, we won't be the first" to defend a path that's been flagged as risky, the disagreement has three possible resolutions: (a) the stakeholder is right and the risk was overstated, (b) the destination is well-trodden but they're pointing at the wrong *path* to get there, or (c) the path actually is genuinely novel/risky. Don't reflexively agree (people-pleasing) or disagree (defensiveness) — verify the claim by checking what real production products are doing in that space. Often the well-trodden path is X (mainstream cloud-hosted, enterprise-tier), not Y (direct-to-publisher developer API), and the reframe is "yes, well-trodden — and the standard path is X, not Y."
**Apply when**: A risk-vs-precedent debate hinges on "industry adoption" or "well-trodden path" claims. Search for the actual production pattern (which providers / hosts / architectures real shipping products use) before settling the question. The right intervention is often not "challenge the precedent" but "verify the path matches the precedent."
