# /pry — accumulated learnings

## Model Choice Is Separable From Inference Path — Weights ≠ Host (2026-05-13)
**Learning**: When a "we can't use $MODEL because $HOST has $PROBLEM (sovereignty / data-residency / outage / pricing)" claim surfaces, decompose into two separable questions: (a) is the model open-weight (freely distributed weights anyone can serve)? (b) which infrastructure providers host those weights in the target region? Open-weight families are served by many hosts across many regions, so the host is a separate decision from the model. The hyperscaler ML platforms (AWS, GCP, Azure) now serve major open-weight families from EU/UK and other residency-sensitive datacenters — making the cloud provider, not the model publisher, the residency anchor, and invalidating many "publisher is in $RESTRICTED_REGION" objections that were valid a year earlier. The reframe sequence when a host-specific constraint appears: find another host serving the same weights in the target region → else self-host (open serving runtimes on cloud GPU) → else drop the model, not the use case.
**Apply when**: Any claim that a model can't be used due to its provider's jurisdiction, data-residency, sovereignty, outage, or pricing. Restate as "what hosts serve these weights, and what are their respective constraints?" — check the major cloud ML catalogs' regional availability before treating the constraint as binding.

## OpenAI-Compatible API Is the De Facto Inference Protocol (2026-05-13)
**Learning**: Most modern LLM hosts now expose the same `POST /v1/chat/completions` request/response shape regardless of underlying model. Adding a new provider adapter is therefore ~20 minutes of work: copy an existing OpenAI-shaped adapter, swap endpoint URL + auth header, done. Treat provider-adapter cost as cheap when estimating effort to add a new model path or evaluate a new vendor — the integration is not the bottleneck.
**Apply when**: Estimating effort to add a new model provider to a routing layer. If the provider exposes an OpenAI-compatible endpoint (most do), budget hours not days. Bias toward "try it" rather than "evaluate it" for adapter-only integrations.

## Router-Middleman Aggregators Take a Margin Slice (2026-05-13)
**Learning**: API-aggregator services that proxy requests across many LLM providers add a per-call markup or commission. Even when they offer regional routing or unified billing, they sit as a margin-bearing layer between the consuming product and the actual host. If unit-economics discipline matters, router middlemen are off the table by design — the choice becomes either direct-to-host integration (own the keys, own the margin) or accept the aggregator's permanent COGS slice. Owned first-party gateways (your own multi-provider routing service hosted on your own infra) are a different category: no markup, but operational cost is yours. Distinguish the two: a router middleman is third-party and margin-bearing; a first-party gateway is owned and operational-only.
**Apply when**: Reviewing the inference-layer architecture and someone proposes "let's just route through $AGGREGATOR for simplicity." Check whether unit economics absorb the aggregator markup, and whether the simplicity gain justifies losing direct provider control (rate-limit visibility, failover policy, cost calc authority).

## Transfer Mechanism vs Provider Data-Use Policy Are Different Concerns (2026-05-13)
**Learning**: Cross-border data flow has two distinct compliance gates that are easy to conflate: (a) is there a lawful transfer mechanism in place (Standard Contractual Clauses, adequacy decisions, processor-on-behalf-of-user framing)? AND (b) does the destination provider's default-tier data-use policy match the product's promise to users (e.g., "we will never use your data for training")? A clean transfer mechanism doesn't help if the destination provider's default terms put user inputs into training. Verify both per provider before claiming "sovereignty solved" or "residency handled."
**Apply when**: Reviewing a third-party inference provider for use in a product with data-protection promises. Check both the transfer mechanism AND the provider's published data-use policy at the tier the product will actually consume. Enterprise tiers often have stricter defaults than developer-tier APIs — confirm which tier is being procured.

## Verify "Well-Trodden Path" Claims Before Agreeing or Disagreeing (2026-05-13)
**Learning**: When a stakeholder argues "this is well-trodden territory, we won't be the first" to defend a path that's been flagged as risky, the disagreement has three possible resolutions: (a) the stakeholder is right and the risk was overstated, (b) the destination is well-trodden but they're pointing at the wrong *path* to get there, or (c) the path actually is genuinely novel/risky. Don't reflexively agree (people-pleasing) or disagree (defensiveness) — verify the claim by checking what real production products are doing in that space. Often the well-trodden path is X (mainstream cloud-hosted, enterprise-tier), not Y (direct-to-publisher developer API), and the reframe is "yes, well-trodden — and the standard path is X, not Y."
**Apply when**: A risk-vs-precedent debate hinges on "industry adoption" or "well-trodden path" claims. Search for the actual production pattern (which providers / hosts / architectures real shipping products use) before settling the question. The right intervention is often not "challenge the precedent" but "verify the path matches the precedent."

## A Permissive Middle Tier Often Exists Between Bare-GPU Self-Host and Heavily-Gated Managed APIs (2026-05-16)
**Learning**: When a "managed inference API X is too restrictive for our use case" claim rules out an entire category, the binary choice isn't "use X or self-host on bare GPU." A permissive middle tier often exists: vendor-direct APIs from the model publisher themselves, secondary hosts that serve the same weights with thinner moderation, and private deployments on managed platforms that defer policy to the consuming product. These typically enforce only the legal floor (illegal content, CSAM, NCII, non-consensual deepfakes) and leave brand-safety/NSFW/political policy to the consuming platform. The cost premium over bare-GPU self-host is often a small absolute number at internal scale, and the operational savings (no GPU management, no cold starts, no model-pinning idle cost) are real. Check this tier before defaulting to self-host on the "moderation is too strict" rationale.
**Apply when**: A model-choice debate rules out a managed API on content-policy grounds. Audit the moderation landscape for that model family at three tiers: brand-name managed (heavy gating), vendor-direct or specialist hosts (legal-floor only), bare-GPU self-host (no platform gating). Compare effective cost at the project's actual volume, not list price.

## Read the LICENSE File Before Claiming "Open Weights" or "Apache 2.0" (2026-05-16)
**Learning**: Vendor marketing language ("truly open weights", "open source release", "free for everyone") often conflates "open weights" (the file is downloadable) with "Apache/MIT-style permissive licensing" (free for any use including commercial). Many releases ship custom community licenses with revenue thresholds, commercial-use carve-outs, market-specific restrictions, or non-commercial-only terms. The actual LICENSE file is the binding document; vendor blog posts, README badges, and conference slides are not. Always cite the LICENSE file URL when recommending a model whose license is load-bearing for the product decision (e.g., when revenue trajectory will plausibly cross a license threshold).
**Apply when**: Recommending a model for use in a commercial product, especially when the recommendation hinges on "it's open-source / Apache 2.0." Pull the actual LICENSE file, read it, and flag any revenue threshold, market carve-out, or non-commercial clause as a Phase 2 blocker with a date-stamped re-review trigger.

## Model-Family Licensing Drift Is a Recurring Pattern (2026-05-16)
**Learning**: "Model family X has always been open-weight" is not a forward-looking guarantee. Vendors routinely shift licensing at major version boundaries — what was Apache-2.0 in v2.0 may go API-only in v2.5 (or non-commercial in v3.0). The shift is often unannounced in release notes and only visible by reading the new version's LICENSE. This pattern has played out across multiple model families in 2024–2026. Always re-verify the license at the specific version under consideration, not the historical pattern of prior versions. The forward-looking question is "what does this vendor's LICENSE say today?", not "what did they release last year?"
**Apply when**: Upgrading a downstream product to a newer version of a model family that's been previously vetted. Treat the version bump as a fresh license review, not a continuation of the prior version's terms.

## Check Moderation Scope by Surface, Not by Provider (2026-05-16)
**Learning**: When evaluating an inference provider's content moderation, decompose by surface: input uploads (file safety scans), input prompts (text-content filtering), model outputs (output classifier), and account-level (post-hoc review, suspension policy). These are independent surfaces with independent policies — a provider's strict upload moderation does NOT imply strict prompt or output moderation, and vice versa. Per-model behavior often varies within a single provider (Provider P's hosting of Model A may have very different moderation behavior than Provider P's hosting of Model B). Two parallel research sources can give conflicting reads on this exact point — verify per-model, not platform-wide.
**Apply when**: Auditing whether an inference provider is acceptable for a content-policy-constrained product. List the four surfaces, ask what runs at each, and test the actual model the product will use rather than relying on the provider's headline policy.

## "Can't Scrape Without Proxies" Claims Usually Conflate IP Block With Fingerprint Block (2026-06-16)
**Learning**: When a scraper is blocked by Google (or any major platform), the block has two independent causes that require different fixes: (1) IP reputation — datacenter CIDRs get blocked immediately; residential/home IPs get much more latitude. (2) Browser fingerprinting — `navigator.webdriver`, WebGL renderer, canvas signature, CDP protocol leaks, plugin list absence. Most Playwright setups only patch HTTP headers, leaving all fingerprint signals exposed. The fix for (1) is proxies; the fix for (2) is a patched browser (camoufox-js, patchright, playwright-stealth plugin). These are separable. On a residential IP with a properly patched browser, Google Maps scraping often works without any proxies. Always decompose before assuming proxy spend is necessary.
**Apply when**: Any "we need proxies to unblock scraping" claim. First verify: (a) what IP type are we running on? (b) what fingerprint signals is the browser leaking? The answer to (b) is almost always "all of them" with vanilla Playwright — and a patched browser (camoufox-js) is free.

## OSM Overpass API Is a Free, Block-Proof Business Discovery Layer (2026-06-16)
**Learning**: OpenStreetMap's Overpass API is completely free, requires no API key, has no bot detection, and returns structured business data (name, address, phone, website, coordinates) for any geographic bounding box. For Philippines specifically: 10-20 dental clinics returned for Quezon City in a live test. PH OSM has good name/address coverage but thin phone/website (~15-20%). The right usage pattern: use Overpass for *discovery* (get business names + addresses), then pass those to a website enrichment scraper for contact data. Public endpoints: overpass.kumi.systems (no rate limit stated), overpass-api.de (10K queries/day), overpass.private.coffee (no rate limit). Overpass QL syntax for business type + bounding box: `node["amenity"="dentist"](lat1,lon1,lat2,lon2);out center;`
**Apply when**: Any project that needs PH (or global) business discovery without paid APIs or Google SERP. Slot this in as a baseline source before any scraping — it always returns something and is legally unambiguous.

## Scraper "Zero Leads" Bugs Are Often Code Path Bugs, Not Just Blocking (2026-06-16)
**Learning**: When a scraper returns 0 or garbage results, the instinct is "we're blocked." But frequently there's also a code bug that converts a partial-block into a total-garbage result. In the Sweeper case: Google Maps loaded the search page (status 200), the feed was not rendered in time (5s timeout), the code dropped into a single-result h1 fallback, grabbed the page heading "Results" as a business name (bypassing the SKIP_LABELS check), and stored it as a lead. The engine returned 1 "lead" named "Results" instead of 0 leads + an obstacle. Separately, when 45 card detail-clicks timed out (3-strike detection missing), the engine ran for 4.5 minutes producing nothing. Both are bugs, not just blocking. Always distinguish: (a) the site blocked me vs (b) the code incorrectly handled the blocked state.
**Apply when**: Reviewing a scraper that returns 0 or junk. Before blaming the target site, trace the actual execution path for the zero/junk result — it's often a missing guard in an error/fallback branch.

## Token-Saving Analyses Are Denominated in the Runtime's Scarcest Resource — Re-Rank When the Runtime Changes (2026-06-19)
**Learning**: A token-reduction lever list is implicitly priced in *one* currency — for a cloud API that's per-token dollars, so the biggest file-read wins top the list. Port the same skill stack to a **local open-weight model** (e.g. a 1M-context open model, locally deployed) and the currency flips to {context-window budget, wall-clock prefill+decode latency, single-GPU serialization} — marginal token cost ≈ 0. The *ranking inverts*: (a) fan-out subagents that ran cheap-and-parallel in the cloud now **serialize on one GPU** → fan-out batching becomes the headline latency ceiling, not a "spiky/optional" lever — and on a TUI with no subagent spawning the sequential fallback fires automatically, so an explicit sequential-fallback contract is the real fix; (b) file-read gating loses its dollar rationale AND its window rationale if the local model also has a large context (a 1M-context local model means the "smaller window crowds out work" argument is FALSE — only prefill-latency remains); (c) output-compression (strip/dedup/truncate) is the one harness-agnostic win — helps tokens, prefill, and window simultaneously; (d) measurement tooling that reads the *cloud vendor's* transcripts doesn't exist on the local runtime — instrument wall-clock + the OpenAI-compatible `usage` block instead; (e) per-skill *model-tier* ceilings are inert on a single-model runtime, but the *intent* (cheap skills shouldn't burn max reasoning) ports onto an effort/thinking-level knob if the local model exposes one.
**Apply when**: Reviewing any token/cost optimization, or porting a fan-out-heavy skill repo to a local/self-hosted model. Ask "what's actually scarce in the TARGET runtime?" before accepting the lever ranking — and verify the target's context window before claiming file-reads crowd it. The cloud→local shift turns the parallelism assumption (cheap concurrent subagents) into its opposite (serialized GPU passes).

## "Do I Still Need $VENDOR?" Splits Into Model / Harness / Subscription — Answer Them Separately (2026-06-19)
**Learning**: When someone says "open model X beats $VENDOR's model, so my $VENDOR subscription is now pointless," the claim bundles three separable decisions: (a) **model quality** — is X actually ahead, or just *near*-parity? Benchmark marketing ("nears the leader", "0.7 pts from the king") usually means *close on short evals but the gap is WIDEST on long-horizon agentic/multi-step tool-use* — which is exactly where a skill stack's heavy arts (autonomous builders, multi-persona QA, go-live reviews) live, so near-parity on aggregate ≠ parity on YOUR workload; (b) **harness** — the agentic loop (skill discovery, subagent fan-out, hooks) is the vendor's *CLI*, not its *model*; you can keep the CLI and point it at the open model via a base-URL override + OpenAI↔vendor proxy, or move to a neutral harness the stack already supports; (c) **subscription/cost** — "drop the sub = self-host the open model free" is usually FALSE: frontier open weights are huge (hundreds of GB VRAM even quantized → a several-thousand-dollar rig or quality-lossy low-bit quant). The real cost-saver is the *hosted* open-model API or its flat coding sub (often 3–6× cheaper per token), NOT local self-host — reserve self-host for sovereignty/offline, not savings. The unlock: a tool-neutral stack means the answer is almost always "yes you can switch the model, keep the harness, go hosted-not-local — but A/B your own heaviest task first because that's where the open model's gap (if any) concentrates."
**Apply when**: Any "should I cancel $VENDOR / switch to open model X" question for an agentic tool stack. Decompose into model/harness/subscription, check whether the benchmark gap lands on the user's actual heavy workload, and steer cost-saving toward hosted-API/flat-sub over local self-host unless data-residency is the real driver. Recommend a cheap empirical A/B (point the existing harness at the hosted open model, run the real load) before any cancellation.

## Deprecated Analytics-API Metrics Are Not Version-Gated — Don't Chase Older API Versions (2026-06-17)
**Learning**: When a platform analytics/insights API starts rejecting a metric with a hard error (e.g. a generic `#100 invalid metric`), two failure classes look identical but behave oppositely: normal *field/schema* changes ARE version-gated (an older API version still serves them), but deprecated *insights/analytics metrics* are NOT — a killed metric errors on EVERY version, so "try an older API version" is a guaranteed dead end for metric deprecations. Three more reusable facts from the same family of APIs: (1) **all-or-nothing batching** — one invalid metric in a comma-joined list fails the WHOLE request so every metric returns empty, masquerading as "silently N/A"; always test metrics individually. (2) **silent date-triggered breakage** — code that "verified valid" before a deprecation date breaks ON the date with no code change; watch the platform's API changelog for the ~90-day notice. (3) **undocumented-replacement dead ends** — when docs describe a conceptual replacement ("Reach becomes Media Viewers") but name no API field, guessing field names wastes time and breakdown params often don't validate (bad values don't error, so you can't introspect valid ones). The reframe that preserves value: pivot dashboards to whatever metric DOES survive as the canonical headline rather than chasing a unique/granular field that isn't exposed.
**Apply when**: Any "this analytics metric stopped working / got killed with no replacement" claim on a third-party platform API. Skip the older-version experiment for metric deprecations, test metrics one at a time, read the changelog for the deprecation date, and reframe toward the surviving metric instead of guessing undocumented field names.

## Refusing an Adequate Alternative — the Wall Is Usually Imported Distrust, Not the Stated Object (2026-06-21)
**Learning**: When a person refuses an obviously-adequate alternative citing a constraint ("X isn't safe / clean / good enough"), the stated constraint is rarely the real wall — it's an imported belief or distrust, often a habit that was rational in another context and is now obsolete. Decompose toward the *belief*, not the object: the load-bearing assumptions ("X isn't safe," "the alternative won't satisfy them") are the seam, and on evidence they're usually false. The strongest lever is to **make the invisible measurable** — find the cheap instrument that *demonstrates* the truth, because for a skeptic a demonstration beats an argument (generalizable: when distrust blocks adoption, reach for the instrument that shows the gap, not the rhetoric that asserts it). Re-scope the time horizon when "not worth it" ignores a permanent payoff that dwarfs the trial period. And always offer a zero-friction parallel fallback so there's an immediate win even while the trust-fix takes time to land. Present options tiered by friction tolerance — best-but-needs-a-behaviour-change → middle-path-that-preserves-the-feel-of-the-old-habit → no-friction-fallback — and let the person pick their own tolerance rather than forcing one answer.
**Apply when**: Any pry where the blocker is a person's refusal or distrust rather than a technical constraint (lifestyle, household, adoption, stakeholder buy-in). Decompose to the underlying belief, look for a cheap measuring instrument that demonstrates rather than argues, and pair the behaviour-change ask with a zero-friction fallback. Output as a friction-tiered menu, not a single verdict.

## Match the Deploy Target to the Workload Class — Existing Team Infra Is a First-Class Deviation Signal (2026-06-21)
**Learning**: A stack-guide's "hosting" row is usually calibrated for ONE workload class (typically a web app: server-rendered API + SPA → a managed PaaS / hyperscaler serverless-containers default). A *different* workload class — GPU model-serving (vLLM, speech STT/TTS, local LLMs) — must not inherit that default by reflex; the guide's own instruction is "evaluate fit." The strongest signal to deviate is **the team already operating a cheaper, better-fit stack for the same workload shape** — existing ops muscle + cost + workload-match outweigh "it's the documented default." A common winning shape for GPU serving: a cheap **always-on dedicated box** (flat monthly; runs the API + DB/pgvector + light local inference; a single box keeps local-file state like a usage ledger valid, deferring any shared-store migration) + **serverless GPU** (per-second, scale-to-zero, sub-200ms cold starts) for bursty/heavy inference + a **CDN/edge** layer (DNS/TLS/DDoS, plus an identity Access-gate on admin paths — friction-free vs an IP allowlist). Two refinements: (1) put durable crown-jewel state on the lowest-company-risk vendor and ephemeral burst on the riskier one, and keep the burst **portable** (OpenAI-compatible containers) so a dead/acquired/price-hiked vendor is a days-long migration, not a catastrophe; (2) **vendor longevity matters only in proportion to lock-in** — a hyperscaler's near-zero death risk is partly cancelled by its high proprietary lock-in, while a bootstrapped-profitable host or a portable serverless-GPU vendor lets you leave cheaply. If regulated, region choice carries a sovereignty axis distinct from residency: an in-jurisdiction-owned host can give sovereignty a foreign-owned cloud's local region cannot (extraterritorial data-access law still reaches it).
**Apply when**: Choosing or challenging a deploy target for any non-web-app workload (GPU serving, batch, data-heavy), especially when the team already runs infra for a similar workload. Decompose "is the prescribed default right?" into cost / ops-fit / workload-match / lock-in / (if regulated) sovereignty, and weight existing team infra heavily before accepting the stack-guide default.

## Name-Ownability Sweep: Four Checks in One Parallel Batch (2026-07-04)

**Learning**: "The name is probably taken" decomposes into four independently checkable claims, all batchable in parallel: (1) package-registry availability via the registry's JSON endpoint (an HTTP 404 means free; the HTML package page often bot-blocks while the JSON endpoint does not); (2) domain status via whois PLUS a direct fetch of the site — a registered domain that refuses connections or serves nothing is a dormant/squatter hold (brokerable), which is a categorically different obstacle from an operating business; (3) same-market collision via web search — a name is only truly blocked by a company in the same market; an identically named business in an unrelated field does not block it; (4) trademark: the misconception "a real dictionary word can't be trademarked" is FALSE — real words are freely trademarkable outside their literal domain, so a word's dictionary meaning only conflicts in its own goods class. Trademark aggregator sites bot-block automated checks; infer from same-class usage plus first principles, then flag a formal registry search as a pre-filing must. Also: registries for some newer TLDs don't publish standard whois — "no data" reads as likely-available but must be confirmed at a registrar.

**Apply when**: Any product/brand naming investigation; run the four checks as one parallel batch before declaring a name blocked or free.
