## Drizzle Migration Operations Across Fresh and Existing DBs (2026-03-19)
**Learning**: Three rules together govern safe Drizzle migrations: (1) **Versioned for persistent, push for ephemeral** — `db:migrate` for production and any persistent DB; `drizzle-kit push` only for throwaway DBs (CI test runs, local dev reset). Mixing on the same persistent DB causes drift, since push doesn't track history. (2) **Seed the journal when transitioning** — Drizzle's `migrate()` stores history in schema `drizzle` table `__drizzle_migrations`; when switching from push to versioned, create the schema/table and insert a row for the baseline migration so the migrator's `created_at` comparison knows what's already applied. Deploy pipelines should auto-stamp the baseline before running `db:migrate` on push-originated DBs. (3) **Guard fixup scripts against fresh DBs** — `ALTER TABLE` fixups crash when the base table doesn't exist yet; always wrap in `SELECT 1 FROM information_schema.tables WHERE table_name='X'` so the fixup only runs when the table is present.
**Apply when**: Any project transitioning from drizzle-kit push to versioned migrations, or operating Drizzle migrations across both fresh and existing databases.

## Supabase RLS Must Use JWT Claims, Not Subqueries (2026-03-28)
**Learning**: RLS policies with subqueries like `WHERE firm_id = (SELECT firm_id FROM profiles WHERE id = auth.uid())` re-execute the subquery per row — O(N) instead of O(1). The fix: store `firm_id` and `role` as custom JWT claims via a Supabase Auth hook, then use `auth.jwt()->'app_metadata'->>'firm_id'` in policies. This is a constant comparison with no subquery. Mark helper functions as `STABLE` not `IMMUTABLE` to avoid stale cache issues within transactions.
**Apply when**: Writing Supabase RLS policies for any multi-tenant app. Always use JWT claims over subqueries.

## Supabase + Drizzle Dual-Access Pattern for Complex Apps (2026-03-28)
**Learning**: For apps with 20+ tables and complex queries (joins, CTEs, aggregations, transactions), use BOTH Supabase JS client and Drizzle ORM. Supabase JS for: auth, realtime, storage, and simple client-side CRUD (benefits from RLS). Drizzle for: server-side queries, reports, transactions, and complex joins (bypasses RLS, so enforce auth in application code). Connect Drizzle via Supabase's connection pooler URL (port 6543, transaction mode), not direct connections (which exhaust limits from serverless functions).
**Apply when**: Starting any Supabase project with more than 15 tables or complex reporting requirements.

## LibreOffice in Docker Needs a Persistent Daemon (2026-03-28)
**Learning**: LibreOffice headless Docker images are 400MB-1.2GB. Each conversion spawns a fresh `soffice` process (1-3 second cold start, 200-500MB memory spike). For production: use Gotenberg (Go microservice wrapping LibreOffice with queuing) or unoserver (keeps LibreOffice as a persistent daemon). Never convert documents synchronously in an API request — always queue conversions. Budget for a small dedicated conversion container — free-tier compute is insufficient for the memory profile.
**Apply when**: Adding document conversion to any project. Never put LibreOffice on a free-tier container.

## Free Tier Limits Change Without Notice (2026-03-29)
**Learning**: Cloud API providers silently change free tier limits without announcements. Always verify current limits against the actual API pricing page at architecture review time, never trust cached or remembered values. Secondary models or tiers may become the practical workhorse when primary tier limits drop.
**Apply when**: evaluating any cloud API free tier during architecture review or probe — verify limits are current, not assumed.

## ML Library Language Ecosystem Compatibility (2026-03-29)
**Learning**: Python ML libraries (speech processing, embeddings, computer vision) often have no JavaScript/WebView implementation path. When a blueprint specifies an ML model for on-device mobile use in a hybrid app, verify the language ecosystem compatibility first. ONNX export + native runtime is the bridge, but adds significant complexity vs. a server-side approach.
**Apply when**: any blueprint or architecture that specifies ML inference on a mobile/hybrid client — verify the model has a JS-compatible runtime or plan for server-side inference.

## Verify TTS Free-Tier Ceilings Against Project Volume Before Committing (2026-03-29)
**Learning**: Cloud TTS free tiers have monthly character ceilings that are easy to exceed in voice-heavy products. Unofficial wrappers around vendor browser endpoints periodically appear in the npm/pip ecosystem promising "unlimited" access — they are unauthorized, fragile, and TOS-grey. Treat them as throwaway prototyping tools, not production dependencies. For production: model the monthly character volume against current verified free-tier limits at probe time, then choose between (a) staying within a verified free tier, (b) committing to a paid plan, or (c) self-hosting an open-weights TTS. Specific package names belong in stack guides, not in this evergreen learning — they decay rapidly when the underlying vendor endpoint changes.
**Apply when**: Selecting a TTS provider for any voice-enabled project. Verify free-tier limits against current pricing pages, never trust cached recommendations.

## Serverless DB + Container Cold Start Warming Strategy (2026-03-29)
**Learning**: Scale-to-zero serverless databases (300-500ms cold start) combined with scale-to-zero compute containers (2-5s cold start at min-instances=0) produce first-request latencies of 4-9 seconds. A health-check cron every 4 minutes during business hours is the free-tier warming strategy.
**Apply when**: deploying any app on serverless DB + serverless compute free tiers where scale-to-zero is enabled.

## iframe Sandbox Defense-in-Depth for AI-Generated Content (2026-03-29)
**Learning**: `sandbox="allow-scripts"` without `allow-same-origin` prevents the classic sandbox escape, but is insufficient alone for AI-generated HTML. LLM output injection (OWASP LLM05:2025) means the generated content itself is the attack surface. Required layers: (1) serve from separate origin, (2) CSP headers `connect-src 'none'` to block exfiltration, (3) server-side DOMPurify sanitization before storage, (4) content safety scan before first render (not async post-publish).
**Apply when**: Any platform serving AI-generated content in sandboxed iframes.

## Auth Libraries With Flat User Models Need Custom Sub-Profile Tables (2026-03-29)
**Learning**: The general rule is "auth library owns identity, don't create parallel user tables" (see `Auth Library Territory` in global-patterns.md). The exception: auth libraries with flat data models have no native support for hierarchical relationships like parent-child accounts. Organization plugins are for SaaS tenancy, not family structures. The correct pattern: auth library handles primary user auth only; subordinate profiles (children, sub-accounts) use a first-party table with scoped session tokens managed entirely by application code (Netflix, Disney+, Roblox pattern).
**Apply when**: Any project using an auth library that needs hierarchical user relationships (parent-child, manager-employee, guardian-ward).

## Scalability Analysis Belongs in /probe, Not /press or /pound (2026-04-02)
**Learning**: Scalability is an architecture concern ("does this design scale?"), not an ops concern ("are we ready to deploy?"). It belongs in `/probe` as a "Scaling Runway" evaluation section that assesses: (1) stateless request handling, (2) database scaling path (pooling → replicas → partitioning → sharding), (3) external dependency scaling (AI provider, CDN, queue), (4) what user count triggers each scaling tier. Present as a table: user range → required change → effort level. `/press` evaluates current state (go-live readiness); `/probe` evaluates future state (design validity). A scaling ceiling is a design flaw, not a deployment gap.
**Apply when**: Running `/probe` on any blueprint or architecture. The scaling runway should be part of every probe report.

## Fraud/Risk Fields Belong on the Event Table, Not the Entity Table (2026-04-25)
**Learning**: When fraud-detection fields (risk score, fraud flag) are derived from per-submission signals (IP, device, timestamp, fingerprint), they must live on the same table as those signals — not on the parent entity. Placing them on the entity when the source data is on the event forces either expensive joins or denormalization drift. The computation belongs where the evidence lives.
**Apply when**: Any blueprint with fraud scoring, risk flagging, or audit fields — verify the scored fields and their source signals are co-located on the same table.

## Cloud Task Queues Should Use Platform IAM Tokens, Not Shared Secrets (2026-04-25)
**Learning**: Using a shared-secret header (e.g., `X-Internal-Secret`) for cloud-task-queue → service authentication is an anti-pattern: secrets rotate manually, they can leak via logs, and they don't integrate with the platform's IAM auditing. The correct pattern is short-lived signed tokens — the task queue attaches a token from a dedicated service identity, and the receiving endpoint verifies it automatically when IAM-protected. Examples: GCP Cloud Tasks → Cloud Run uses `oidcToken: { serviceAccountEmail, audience: ENDPOINT_URL }`; AWS SQS → Lambda uses IAM execution roles; Azure Service Bus → Functions uses managed identity.
**Apply when**: Any architecture using a cloud task queue to invoke a service endpoint. Always specify platform-native IAM tokens; never use header-based shared secrets.

## Rate Limiting Needs Shared State From Day 1 on Auto-Scaling Platforms (2026-04-25)
**Learning**: In-memory rate-limit middleware silently breaks when a service scales beyond 1 instance — each instance maintains its own counter, so a client hitting multiple instances bypasses the limit by a factor of `max-instances`. This is not a scaling concern to defer; it's a Day 1 correctness issue for any app deployed on auto-scaling platforms (serverless containers, FaaS, managed compute). The fix: a shared store (Redis, Memcached, DynamoDB) backing the rate-limit middleware.
**Apply when**: Any blueprint with rate limiting on an auto-scaling platform. Always spec shared-store rate limiting, never in-memory.

## SMS Gateway OTP APIs May Reject Pre-Generated Codes From Auth Libraries (2026-04-25)
**Learning**: Some SMS gateways have their own OTP generation APIs and reject codes pre-generated by your auth library — the formats, character sets, or expiry semantics don't match. Two paths: (A) use the auth library's code generation + the gateway's plain SMS message API (e.g., `Your OTP is: ${code}`) — simplest and most reliable; (B) use the gateway's OTP API and implement a custom flow outside the auth library. Path A is strongly preferred unless carrier-grade delivery features (delivery receipts, network priority routing) are required.
**Apply when**: Integrating any auth library's phone OTP feature with a regional SMS gateway. Verify the gateway accepts pre-generated codes before committing to the auth library's native flow.

## Real-Time Default Should Be SSE; Managed Pub/Sub Is the Upgrade Tier (2026-05-10)
**Learning**: WebSocket pain ≠ real-time pain. Managed pub/sub services (Ably, Pusher, PubNub) solve the *WebSocket* class of bugs (reconnection, presence, multi-region delivery), but the typical web-app workload — fanout notifications, live list updates, modest concurrency, no chat/multiplayer/financial-ticker requirements — does not need WebSockets at all. **Server-Sent Events (SSE)** — the W3C `text/event-stream` standard, browser-native via `EventSource`, with built-in auto-reconnect and `Last-Event-ID` resume — covers that workload with zero vendor lock-in, $0 cost, no SDK, plain HTTP that works through every proxy/CDN, and a server implementation that's a long-lived response writer most modern frameworks handle natively. Managed pub/sub's differentiated value props (built-in presence, durable history with replay, multi-region delivery guarantees, true bidirectional channels) only matter for a minority of projects. Default to SSE; reserve managed pub/sub for the upgrade tier when presence/history/multi-region is a genuine requirement. Mirrors the existing database tier pattern (commodity default, managed upgrade). One server-side caveat: SSE is awkward on platforms with long-lived-connection limits (some serverless container runtimes, edge workers) — projects deploying there should evaluate the managed tier earlier.
**Apply when**: Reviewing or drafting a Pattern's real-time layer. Ask whether the workload is fanout-style (SSE) or presence/history-style (managed pub/sub). Don't auto-reach for a managed pub/sub service from reflex; the reflex usually conflates "I need real-time" with "I need WebSockets". Stack-guide entries that list a managed real-time service as the sole option should be split into a tiered entry with SSE as default and the managed service as upgrade.

## OAuth `state` Must Be Browser-Bound, Not Just Signed (2026-06-21)
**Learning**: A signed-and-unexpired OAuth `state` parameter is necessary but NOT sufficient for CSRF protection on the authorization-code callback. A purely cryptographic state proves "this server issued some state," not "this state belongs to the browser now completing the flow" — an attacker who obtains any valid state can replay it to fixate a session on a victim. Bind the state to the initiating browser: at login-redirect time set a short-lived `HttpOnly`/`Secure`/`SameSite` cookie carrying the same nonce; on callback require BOTH (a) the returned state validates (signature + expiry) AND (b) it matches the browser's cookie via a constant-time compare. Reject when the cookie is absent or mismatched — that is the exact case that catches a forged or cross-browser-replayed state — and delete the cookie on success. The same principle applies to any redirect-based handshake secured by a state/nonce round-trip.
**Apply when**: Reviewing or designing any OAuth2 / OIDC authorization-code flow, or any redirect-based callback secured by a `state`/nonce. Verify the state is bound to the browser (cookie or server-side session), not merely signed.

## Ephemeral Fallback Signing Secrets Break Silently Across Workers (2026-06-21)
**Learning**: When a session/JWT/HMAC signing secret is unset and the code "helpfully" falls back to a process-generated random key, it passes single-process dev testing and then fails silently in production two ways: (1) under multi-worker / multi-instance / autoscaled deployment each process holds a different key, so a session minted on worker A fails validation on worker B → users are randomly and intermittently logged out (load-balancer-dependent, near-impossible to reproduce locally); (2) every restart or redeploy mints a new key, invalidating all live sessions at once. A silent ephemeral fallback converts a missing-config error into a heisenbug. At minimum, emit a loud ONE-TIME warning at startup when the fallback fires so the misconfiguration is visible in logs; better, fail closed in production and permit the ephemeral key only in an explicit dev/local mode. Any horizontally-scaled signer needs a persisted, shared secret.
**Apply when**: Reviewing auth/session/token-signing setup for any service that may run more than one process (multi-worker servers, autoscaling, serverless, blue-green deploys). Flag any `secret or random()` fallback that neither warns loudly nor fails closed.

## Publish-Once Guarantees: the Ledger Key and Write Ordering Are Where They Break (2026-07-04)

**Learning**: Two defects recur in otherwise careful exactly-once designs for external side effects, both invisible until production: (1) a uniqueness key that omits the tenant dimension — e.g. (itemSlug, destination) without tenantId — turns cross-tenant slug reuse into a silently LOST action (ON CONFLICT DO NOTHING swallows it), worse than the duplicate it guards against; (2) writing the ledger row AFTER the external call means a crash between external success and ledger write duplicates on retry. The guarantee requires claim-before-side-effect: txn 1 inserts the row with status='in-progress' (ON CONFLICT DO NOTHING; no row returned ⇒ already claimed ⇒ no-op), the external call runs outside any transaction, txn 2 updates with the external id; stale in-progress rows reconcile via a verify/list-recent step. State the delivery model explicitly — "exactly-once against external HTTP does not exist; we run at-least-once drain + at-most-once side effect per claim" — or builders trust the wrong invariant.

**Apply when**: Reviewing any system that must perform an external side effect exactly once per logical item (payments, emails, posts, webhooks, provisioning). Check the uniqueness key for the tenant dimension and the ledger write ordering first.

## Self-Hosted Single-Box Deployments Need a Resurrection Chain and Out-of-Band Detection (2026-07-04)

**Learning**: An application can be genuinely downtime-tolerant (durable queues, catch-up drains, upstream-native scheduling) and still suffer unbounded outages because the HOST layer was never specified. Five links, each defaulting to off: host-boot auto-start of the VM/runtime (e.g., Task Scheduler for WSL2), init system enabled inside it (systemd + container engine), the ingress tunnel installed as a host-level service so it survives VM restarts, container restart policies (unless-stopped), and a memory cap sized so the host never kills the VM. And every alert channel that originates from the box dies with it — a dead-man's switch (external service expecting a periodic ping) is one cron line and turns "discovered Monday" into "paged in minutes." Restore discipline for side-effecting systems: a restored dump forgets everything since the dump, so the restore runbook must reconcile against external ground truth (list-recent from the destination platforms) before the scheduler resumes, or the restore itself causes duplicate side effects; the encryption key file must be backed up separately from the dumps it protects.

**Apply when**: Any self-hosted or single-box deployment review. Walk the five-link chain, demand the dead-man's switch, and check the restore runbook for reconcile-before-resume and key/dump separation.

## Analytics-Consumed Fields Belong in Typed Columns; Self-Tuning Loops Need Fixed Horizons and Sample Gates (2026-07-04)

**Learning**: The hybrid storage pattern — raw external payloads in jsonb (schemas differ, fields deprecate), comparables promoted to typed NULLABLE columns — is load-bearing wherever aggregation happens: Postgres keeps no planner statistics on jsonb internals, percentile/median functions need typed columns, and nullable columns encode "not available ≠ zero" for free. For self-tuning loops fed by such metrics: record outcomes at fixed maturity horizons (e.g., 48h and 7d, horizon stored on the row) rather than "next day," compute baselines over rolling windows, and gate any automated rebalancing on a minimum sample count — medians over tiny samples thrash the system they steer.

**Apply when**: Reviewing metrics ingestion from external platforms or any learn/optimization loop; promote the group-by axes to columns and check horizon + sample-size discipline.
