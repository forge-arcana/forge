**Key revisions**:
1. (Q1) Skills go to `.agents/skills/<name>/SKILL.md`, not `<project>/skills/<name>/`. The Agent Skills specification (Anthropic, December 2025; cross-tool by January 2026) standardized this path. Codex reads it primary, Claude/Gemini/DeepSeek discover it as fallback.
2. (CLAUDE.md /pry pass) The Claude bootstrap is **a 1-line file** (`@AGENTS.md`), not a symlink. Anthropic officially recommends the `@-import` mechanism for this exact scenario; recursive up to 5 hops; cross-platform (no Windows symlink fragility); Anthropic-documented (lower risk of upstream breakage). Symlink works as a fallback but is not the recommended path.

## 1. Goal

Prove that Claude Code can run a **generic forge** — a forge with no per-vendor adapters, no vendor-specific build paths, no Claude-coupled tooling in `core/`. The forge emits the cross-tool standard (`AGENTS.md` + `skills/`) once. Claude consumes it via two thin symlinks. Other tools (Codex, Cursor, Bob, Amp, Factory) consume it natively without modification.

The MAXIMA branch's previous direction — per-vendor adapters (`adapters/claude-code/`, `adapters/bob/`, `adapters/antigravity/`) — was a softer flavor of vendor lock-in. Building tooling per-vendor multiplies the matrix and silently re-couples the build pipeline to whichever manifests we maintain. Path A v2 rejects that approach.

The reframe: **what looked like an adapter system was a workaround box for Claude's OAuth race bug, plus two reference docs**. None of it was forge-functionality-required. The whole "adapter" architecture rested on a misunderstanding of what `adapters/claude-code/` actually contained.

## 2. Background — what `/pry` found

The /pry session decomposed the wall *"Claude can't run a forge that emits only AGENTS.md"* into seven assumptions. Six were FALSE, one was TRUE-but-trivially-bypassable. Specifically:

| Assumption | Verdict |
|---|---|
| Claude doesn't read AGENTS.md natively | TRUE → bypassed by `ln -sf AGENTS.md CLAUDE.md` (Anthropic-supported behavior) |
| Slash commands must live at `~/.claude/skills/` | TRUE → bypassed by `<project>/.claude/skills/<name>/SKILL.md` (project-scoped, no global) — and that path is itself reachable by symlink |
| AGENTS.md isn't expressive enough | FALSE — plain markdown, no schema, accepts any structure |
| `.claude/settings.json` required for forge | **FALSE** — re-evaluated. Inspection of `adapters/claude-code/scripts/` showed every script is a workaround for [anthropics/claude-code#43392 + #24317](https://github.com/anthropics/claude-code/issues) (OAuth refresh-token race, WA-001). The two refs (`claude-code-settings.json`, `claude-code-rules.md`) are user-setup descriptors, not runtime artifacts. **None of it is forge-required.** |
| Subagent spawning needs Claude config | FALSE — `Task` tool is built-in |
| AGENTS.md isn't actually universal | FALSE — 40,000+ projects, Linux Foundation steward, native in Codex/Cursor/Amp/Factory/Bob |
| Global cross-project commands need Claude paths | TRUE → bypassed by accepting project-scoped commands or one-time global symlink |

**The architectural conclusion**: `adapters/claude-code/` is not a vendor adapter — it's a bug-workaround box with descriptive references. Calling it an "adapter" created a false commitment to multi-vendor build paths. The whole concept dissolves when you read what's actually in the directory.

## 3. Target architecture — Generic Forge v2 (revised)

```
forge/
├── core/                          ← single source of truth (no change)
│   ├── rules/
│   │   ├── forge-governance.md
│   │   └── development-discipline.md
│   ├── skills/
│   │   └── <art>/SKILL.md         ← tool-neutral skill bodies (with inline `<!-- model: opus -->` for Claude)
│   ├── scripts/                   ← tool-neutral runtime scripts (read FORGE_PATH env var)
│   └── forge.schema.yaml          ← retire (no adapters → no contract to satisfy)
│
├── bin/
│   └── forge-build                ← rewritten: agnostic, no --adapter argument

├── build/                         ← gitignored, regenerated by build
│   ├── AGENTS.md                  ← thin pointer file: <150 lines, <32 KiB
│   ├── .agents/skills/<art>/SKILL.md   ← OPEN AGENT SKILLS STANDARD path
│   └── scripts/                   ← runtime helpers from core/scripts/
│
├── claude-helpers/                ← renamed from adapters/claude-code/ — clearly NOT an adapter
│   ├── README.md                  ← "Claude bug workarounds + bootstrap. Optional. Forge core does not require these."
│   ├── scripts/                   ← WA-001 OAuth race workarounds (delete when upstream ships fix)
│   │   ├── agent-preflight.sh
│   │   ├── agent-token-warmup.sh
│   │   ├── agent-token-scheduler.sh
│   │   ├── install-token-hook.sh
│   │   ├── user-agent-preflight.sh
│   │   └── sync-workaround-side-effects.sh
│   ├── refs/                      ← descriptive only
│   │   ├── permissions-template.json   ← renamed from claude-code-settings.json
│   │   └── auto-allowed-bash.md        ← renamed from claude-code-rules.md
│   └── bootstrap.sh               ← ONE symlink: ln -sf AGENTS.md CLAUDE.md (+ optional WA-001 hook)
│
├── learnings/                     ← unchanged
├── memory/                        ← unchanged
├── presentation/                  ← unchanged
├── .agents/skills/                ← forge's OWN skills, regenerated by build (forge dogfooding)
│   ├── forge/                     ← thin bootstrap so /forge is discoverable on fresh clone
│   └── (purge stays at .claude/skills/ — forge-internal, intentionally Claude-coupled)
├── CLAUDE.md                      ← 1-line file: `@AGENTS.md` (forge dogfoods its own claude-helpers/bootstrap.sh)
└── AGENTS.md                      ← top-level pointer file (<150 lines)
```

**What changed vs. v1**: skill location is `.agents/skills/<name>/SKILL.md` (Open Agent Skills standard, December 2025), not `<project>/skills/`. This eliminates the second symlink — `.claude/skills/→../skills` is no longer needed because Claude Code (and Codex, Gemini, DeepSeek, Bob, Cursor) all discover `.agents/skills/` natively.

### What `claude-helpers/bootstrap.sh <project>` does (revised — `@-import`, not symlink)

```bash
# bootstrap.sh — opt-in Claude Code bridge for projects that consume forge.
# Idempotent. Run once per project. Cross-platform (no symlink dependency).

PROJECT="${1:?usage: bootstrap.sh <project-path>}"

# 1. Create a 1-line CLAUDE.md that imports AGENTS.md.
#    This is Anthropic's officially documented pattern — the @-import is recursive
#    up to 5 hops, so AGENTS.md can chain other files via @path syntax.
#    The .agents/skills/<name>/SKILL.md files are discovered by Claude natively
#    via the Open Agent Skills fallback path — no separate symlink needed.
echo '@AGENTS.md' > "$PROJECT/CLAUDE.md"

# 2. (optional) install WA-001 hook if user opts in (WSL2-only Claude OAuth race fix)
if [[ "${INSTALL_WA001:-no}" == "yes" ]]; then
  bash "$(dirname "$0")/scripts/install-token-hook.sh"
fi

echo "Forge bootstrapped for Claude Code at: $PROJECT"
echo "Other tools (Codex, Cursor, Gemini, DeepSeek, etc.) use AGENTS.md + .agents/skills/ directly — no bootstrap needed."
```

### What `bin/forge-build` does (rewritten)

```
forge-build [--deploy <project>] [--dry-run]

  Reads:
    core/rules/*.md
    core/skills/*/SKILL.md
    core/scripts/

  Emits to build/:
    AGENTS.md                          ← thin pointer file (<150 lines, <32 KiB)
    .agents/skills/<art>/SKILL.md      ← Open Agent Skills standard path
    scripts/                            ← copied from core/scripts

  --deploy <project>:
    cp build/AGENTS.md          <project>/AGENTS.md
    cp -r build/.agents          <project>/.agents
    cp -r build/scripts          <project>/scripts
```

No `<adapter>` argument. One build. One output. Same shape regardless of which agent will consume it. Skills land at the cross-tool standard path; AGENTS.md is the universal entry point.

### AGENTS.md composition rules (tightened — <150 lines, <32 KiB)

Codex caps `project_doc_max_bytes` at 32 KiB by default. Best-practice target is <150 lines (<371 line upper bound). AGENTS.md must follow **progressive disclosure** — point to detailed files, don't inline. Large AGENTS.md triggers immediate context compaction in some harnesses.

`build/AGENTS.md` is composed by concatenating in order:

1. **Header** — project name, one-paragraph brief, links to forge spec
2. **Rules pointers** — for each `core/rules/*.md`, emit the FIRST PARAGRAPH + a link to the full doc (not the full content). Reader follows links on demand (progressive disclosure)
3. **Model-hint guidance** (one line) — *"Skills marked with `<!-- model: opus -->` benefit from a deep-reasoning model. On Codex/DeepSeek/Gemini, switch via `/model` before invoking these skills."*
4. **Skill registry** — table of all skills with description + TRIGGER + standard path:

   ```markdown
   ## Skills

   | Skill | Description | Trigger | Body |
   |---|---|---|---|
   | /probe | Senior solutions architect | architecture review | [.agents/skills/probe/SKILL.md](.agents/skills/probe/SKILL.md) |
   | /poke | Staff engineer code review | code quality / tech debt | [.agents/skills/poke/SKILL.md](.agents/skills/poke/SKILL.md) |
   | ... | | | |
   ```

5. **Footer** — links to forge governance, lineage notes, version (one line each)

Hard rule: **AGENTS.md never embeds skill bodies**. Skill bodies live in `.agents/skills/<name>/SKILL.md` (one file each). Tools auto-discover `.agents/skills/` per the Open Agent Skills standard; AGENTS.md is the registry for tools that read inline. Both interaction models work from the same artifacts. Target file size: <30 lines of header + ~30 row table + <20 lines of footer ≈ 80 lines total, well under the 150-line target.

## 6. Validation criteria

The pivot is complete when:

1. ✅ `bin/forge-build` runs with no arguments and produces `build/AGENTS.md` + `build/skills/` + `build/scripts/`
2. ✅ `bash claude-helpers/bootstrap.sh <empty-test-project>` creates `<project>/CLAUDE.md → AGENTS.md` symlink and `<project>/.claude/skills → ../skills` symlink, and Claude Code reads them correctly (verify by invoking a slash command in the test project)
3. ✅ The same `AGENTS.md` produced by the build loads on `codex` CLI without modification (smoke test — does not need to run end-to-end, just needs to be parsed without error)
4. ✅ All forge skills still work in this conversation's Claude Code session after the rebuild (the forge repo dogfoods itself)
5. ✅ The `/forge` cycle still runs end-to-end (this is the load-bearing skill — if it works, the build is sound)
6. ✅ No `adapters/` directory exists in the tree
7. ✅ `claude-helpers/README.md` clearly states: *"Optional. Forge core does not require these. Delete when upstream ships the OAuth race fix."*

## 7. Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Claude Code's symlink behavior changes (Anthropic stops following symlinks) | Low — would break thousands of existing setups | Pin claude-helpers to a known-working state; document the assumption; have a fallback bootstrap that copies AGENTS.md → CLAUDE.md instead of symlinking |
| AGENTS.md spec evolves incompatibly | Medium — spec is young | Keep `core/skills/<name>/SKILL.md` as the durable artifact; AGENTS.md is composed; regenerate when spec changes |
| `bin/forge-build` rewrite introduces regressions | Medium | Validation step #4 — dogfood the build immediately; if `/forge` cycle breaks, revert |
| WA-001 fix ships from Anthropic mid-rewrite | Low — they've been silent for months | Watch the issues; deletion of `claude-helpers/scripts/` is a separate small commit when ready |
| `forge-build` deploy step breaks user membranes mid-pivot | Medium | Don't run `--deploy` against `~/.claude/skills/` until the dogfood-test commit (#6) confirms parity |

## 8. Open questions — RESOLVED via second `/pry` pass (2026-05-09)

1. **`.claude/skills/forge/` and `.claude/skills/purge/` in the forge repo** — RESOLVED. With the Open Agent Skills standard (`.agents/skills/`), forge's own self-bootstrap moves to `.agents/skills/forge/`. `/purge` stays at `.claude/skills/purge/` because it's intentionally Claude-coupled (forge-internal, never deployed to user membranes — the design choice predating the pivot stays).

2. **`forge-path:` line in user CLAUDE.md** — RESOLVED. Convert to `FORGE_PATH` env var in Phase A (commit #4). It's a small mechanical change touching the scripts that read it. Not deferral-worthy.

3. **Skill model hints** — RESOLVED via /pry on Codex/DeepSeek/Gemini conventions:
   - **No cross-tool standard exists** for per-skill model selection. Codex CLI uses runtime `/model` slash command (session-level, not per-skill). DeepSeek TUI: same. Gemini CLI: same. All three have `name + description` skill frontmatter with no model field.
   - **Decision**: keep `<!-- model: opus -->` HTML comment inline in SKILL.md frontmatter (Claude reads, other tools ignore harmlessly).
   - **Add** a one-line guidance in AGENTS.md prose: *"Skills marked with `<!-- model: opus -->` benefit from a deep-reasoning model. On Codex/DeepSeek/Gemini, switch via `/model` before invoking."*

4. **AGENTS.md size** — RESOLVED. My initial <50KB target was 50% over Codex's hard cap.
   - **Hard limit**: 32 KiB (Codex `project_doc_max_bytes` default)
   - **Target**: <150 lines (best practice consensus)
   - **Upper bound**: 371 lines
   - **Pattern**: progressive disclosure — AGENTS.md is a thin pointer file with a registry table. Detailed governance rules live in `core/rules/*.md` and AGENTS.md emits ONLY the first paragraph of each + a link. Skill bodies stay in `.agents/skills/<name>/SKILL.md`.

5. **Forge's own working tree during pivot** — RESOLVED (was always the answer): implement on a feature branch (`MAXIMA-build-rewrite`) off MAXIMA. Don't rewrite `bin/forge-build` directly on MAXIMA — branch, validate (commit #7 dogfood test), merge back. Standard branching practice; called out explicitly here because the meta-tool dogfooding itself has a circular dependency that's easy to forget.

### NEW open question discovered in this `/pry` pass

6. **Open Agent Skills spec evolution** — the spec is young (Anthropic Dec 2025; cross-tool adoption Jan 2026). Frontmatter fields, discovery paths, and invocation conventions may evolve incompatibly during 2026. Mitigation: pin SKILL.md frontmatter to the published spec's required fields only (`name`, `description`); avoid optional fields until they're stable. Watch the [agentskills.io](https://agentskills.io/) spec page for changes.

## 9. What we're NOT doing in Phase A

- Inventing a "global membrane" replacement for cross-project skills (defer; project-scoped is sufficient for now)
- Adding explicit per-tool support (Cursor, Codex, Gemini, DeepSeek, etc.) — they all consume AGENTS.md + `.agents/skills/` natively per the Open Agent Skills standard; no per-tool work needed
- Touching the `core/skills/<name>/SKILL.md` bodies themselves (the bodies are universal; only the build pipeline and skill location change)
- Removing WA-001 (still active until Anthropic ships the fix)
- Auditing all skills for residual Claude-isms in their bodies (Phase B work — should already be clean from the de-Claude pivot's Phase 1)
